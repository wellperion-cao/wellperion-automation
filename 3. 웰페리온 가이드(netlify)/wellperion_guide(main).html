③ 3대 원칙 프레임워크 v1.1

정책 SSOT · 모든 AI C-Level 프롬프트에 필수 포함

모든 AI C-Level은 어떤 업무든 이 3가지를 기준으로 한다. 결과물 품질(PSP) · 작성·경로 4원칙 · 시작과 끝. 세 축을 다 통과해야 완성.

🎯PSP — 결과물 품질 기준

표현 원칙: 가독성 · 명확성 · 직관성. 한 번에 이해할 수 있도록 핵심만 남긴다.

최종 판단 기준. "단순한가 → 구조가 있는가 → 의도가 느껴지는가" 세 질문에 모두 Yes여야 완성.

📐작성·경로 표준 4원칙

하나의 진실 (SSOT) — 실무진과 AI C-Level이 한곳에 데이터를 모아두고 업데이트·학습하는 공통 데이터 원본. 중복 정보는 한 곳에만 병합.

  1. 기획 정보 → Start DB
  2. 실행 결과 → 아카이브 DB
  3. 최종 통과(회장·GM·실무진 승인)된 결과물 → 종착지로 이동

속성은 메타데이터만

필터·정렬·대시보드용. 본문은 속성에 안 넣음.

연결은 관계로

복붙 대신 Relation. 원문은 토글로 맨 위.

보고는 문서 구조로

H1/H2 + 표. PDF 인쇄 시 안 깨짐.

큰 것 → 작은 것

목적·구조 먼저 → 실행·증빙 나중.

🔄시작과 끝 — 업무 흐름의 명확성

  1. 모든 업무에는 시작점과 종착점이 있어야 한다
  2. 시작: 기획DB 승인완료 → 착수 (장애·긴급은 예외)
  3. 끝: 결과 보고 → 승인 → 유지보수 전환 → 종착지 정립 = Done

🧠1. Claude Code 핵심 가이드 (11항목)

1-1. CLAUDE.md — 프로젝트 두뇌

  • 세션 시작 시 자동 로드. 프로젝트 루트에 위치.
  • 200줄 이하 유지 (5,000→700 토큰 절약). 서술형 제거, 표/리스트 구조화.
  • 상세 지식은 스킬 파일(docs/)로 분리. 이미 결정된 내용(아키텍처·규칙·빌드 명령)을 명시.

1-2. Plan Mode (설계 모드)

Shift+Tab 2번으로 진입. 코드를 바로 쓰지 않고 먼저 계획을 세운다. 복잡한 작업·아키텍처 결정·대규모 리팩토링 전 활용.

1-3. Task-Do-Verify 패턴

  1. Task: 작업 정의 및 범위 설정
  2. Do: 실행
  3. Verify: 결과 검증

1-4. Context Rot (컨텍스트 부패)

오래 쓰면 출력 품질이 무너지는 구조적 현상. 4가지 원인:

  1. 어텐션 희석 — 문맥이 길어질수록 중간 정보를 못 떠올림
  2. 명령 충돌 — 누적된 지시문이 모순
  3. 토큰 예산 압박 — 시작 파일이 작업할 토큰을 잡아먹음
  4. 관련성 미스매치 — 무관한 정보까지 로딩

실전 임계: 한 파일이 2,000~3,000 토큰(약 1,500단어)을 넘으면 점검 신호.

1-5. Hooks (훅 시스템)

도구 호출 전후 자동 실행 스크립트. PreToolUse(호출 직전) / PostToolUse(호출 직후). 로그 전처리(FAILED·ERROR만 추출), 파일 재읽기 방지, 패턴 학습에 활용.

1-8. MCP (Model Context Protocol)

외부 도구와 Claude Code를 연결하는 프로토콜. 각 MCP 서버는 도구 스키마를 매 메시지마다 로딩하므로 토큰 비용이 크다.

1. 검증 루프 — 지켜보지 않아도 알아서 일하게

앞으로 모든 작업을 "검증 루프" 방식으로 진행해줘.
1. 작업 시작 전에 "무엇이 나오면 성공인지" 성공 기준을 먼저 정의해 알려줘.
2. 작업을 실행한 뒤, 네가 직접 결과를 확인해.
3. 실패하면 에러 원인을 분석해 고치고 다시 시도해.
4. 성공할 때까지 반복하되, 3회 시도 후에도 안 되면 진행 상황을 정리해 보고해.
5. 중간 과정은 생략하고 최종 결과만 보고해.
이 방식을 이번 세션 내내 기본 작업 규칙으로 적용해줘.

2. 검증 루프 영구화 (Skill로 패키지화)

방금 적용한 검증 루프를 재사용 가능한 Skill로 만들어줘.
- 스킬 이름: verification-loop
- 앞으로 모든 작업에서 이 스킬이 자동 적용되도록 설정해줘.
- 작업 중 반복되는 실패 패턴이 발견되면 이 스킬을 스스로 보완·업데이트해줘.

3. 에러 자가복구 루프

앞으로 에러가 발생하면 다음 순서로 스스로 처리해줘.
1. 에러 메시지를 끝까지 읽고 — 무슨 에러인지 / 어느 파일·어디서 났는지 / 어떻게 고치는지 파악해.
2. 파악한 해결책을 바로 실행해 수정해.
3. 최대 3회까지 자가 수정을 시도해.
4. 3회 후에도 해결되지 않으면 그때 정확한 상황(에러 내용·시도한 것·막힌 지점)을 보고해.
그리고 우리 코드에서 에러를 출력하는 부분은, 앞으로 "에러 종류 / 위치(파일·줄) / 해결 방법"이 함께 담기도록 점진적으로 개선해줘.

4. 폴더별 CLAUDE.md 분산 정비

우리 프로젝트를 "폴더별 작업 매뉴얼" 구조로 정비해줘.
거대한 단일 CLAUDE.md 하나에 모든 규칙을 담는 대신, 작업 폴더마다 그 폴더 작업에만 해당하는 규칙을 담은 CLAUDE.md를 두는 방식이야.
1. 먼저 어떤 폴더에 전용 CLAUDE.md가 필요한지, 각 폴더에 어떤 규칙이 들어가면 좋을지 목록으로 제안해줘.
2. 내가 승인하면 그때 각 폴더에 CLAUDE.md를 생성해줘.
루트 CLAUDE.md에는 전사 공통 규칙만 남기는 방향으로.

5. 역할 기반 에이전트 점검 (split-brain 방지)

우리 AI 에이전트(7 C-Level) 구성이 "역할 단위로 완결되어 있는지" 점검해줘.
- 한 업무를 여러 에이전트가 조각내어 나눠 가져 서로 맥락을 모르는 split-brain 구조가 있으면 찾아내줘.
- 각 에이전트가 자기 담당 영역 안에서는 모든 것을 완결적으로 책임지는 구조인지 확인해줘.
- 에이전트에게 일을 위임할 때 항상 전체 컨텍스트(기획·목적·톤·대상)를 함께 넘기는지 확인해줘.
문제가 있으면 재설계 방안을 제안하고, 실제 수정은 내 승인 후 진행해줘.

6. 메모리 정착 — 세션이 끊겨도 학습 유지

이 프로젝트에서 반복되는 결정·규칙·내 작업 스타일·자주 하는 요청을 파악해서, 다음 세션에도 이어지도록 메모리에 정리해 저장해줘.
